# 얼굴 익명화 사이드 프로젝트 실행 계획 (Lightweight)

## 0. 프로젝트 목적
- 기존 완료한 팀프로젝트(이미지 기반 얼굴 익명화)를 사이드 프로젝트로 확장한다.
- 목표는 `학습 + 배포 경험`이며, 논문급 고도화보다 `작동하는 제품`을 우선한다.
- 핵심 3축:
1. 동영상 처리 확장
2. 성능 개선(체감 속도/안정성)
3. 실제 배포

## 1. 성공 기준 (MVP)
- 사용자가 웹에서 `동영상 파일 업로드` 후 익명화 결과를 다운로드할 수 있다.
- 720p 기준 처리 속도가 초기 버전 대비 개선된다(예: FPS 또는 총 처리시간 20%+ 개선).
- 퍼블릭 URL로 접속 가능한 데모 서비스 1개가 운영된다.
- README만 보고 팀원이 로컬 실행/배포 재현 가능하다.

## 2. 범위 정의
### 포함 (In Scope)
- 비디오 파일(`.mp4`) 입력 처리 파이프라인
- 기본 성능 개선(추론 호출 최적화, 프레임 스킵, 리사이즈 전략)
- 웹 배포(백엔드 API + 간단한 프론트)
- 최소 모니터링(로그, 에러 확인)

### 제외 (Out of Scope, 이번 사이클)
- 모바일 네이티브 앱(iOS/Android) 정식 출시
- 실시간 대규모 동시접속 처리
- 완전 자동 MLOps 파이프라인

## 3. 역할 분담 (목표 기반)
- 본인(배포 경험 목표):
  - 인프라/배포 오너
  - CI/CD, 환경변수, 운영 문서
  - 릴리즈 관리
- 팀원(개발 경험 목표):
  - 비디오 처리 기능 개발 오너
  - 성능 개선 실험 및 코드 리팩터링
  - 테스트 케이스 작성
- 공동:
  - 요구사항 확정, 데모 시나리오, 회고

## 4. 4주 로드맵 (너무 헤비하지 않게)

## Week 1 - 비디오 MVP
목표: 이미지 파이프라인을 비디오 파일 처리로 확장
- 작업:
  - `process_video(input_video, reference_image, output_video)` 추가
  - 프레임 읽기/쓰기(OpenCV VideoCapture/VideoWriter) 구성
  - FPS/해상도 유지 정책 확정
  - CLI 또는 API 엔드포인트(`/process-video`) 추가
- 완료 조건:
  - 샘플 영상 1개를 익명화해 결과 파일 생성 성공

## Week 2 - 성능 1차 개선
목표: “느리다” 체감 줄이기
- 작업:
  - 벤치마크 스크립트 작성(처리시간, FPS, 프레임 수 기록)
  - 빠른 개선 3종 우선 적용:
1. 입력 프레임 리사이즈(예: 1080p -> 720p)
2. 프레임 스킵(예: 2~3프레임마다 탐지)
3. 기준 인물 임베딩 캐시 재사용 강화
  - 품질/속도 트레이드오프 파라미터를 설정 파일로 분리
- 완료 조건:
  - 기준 영상에서 초기 대비 성능 개선 수치 확보

## Week 3 - 웹 MVP + 배포
목표: 외부에서 접속 가능한 데모 제공
- 작업:
  - 백엔드: FastAPI에 비디오 업로드/처리/다운로드 API 정리
  - 프론트: 매우 단순 업로드 UI(진행 상태 + 결과 링크)
  - 배포:
1. 백엔드: Render/Railway/Fly.io 중 1개
2. 프론트: Vercel/Netlify 중 1개
  - 스토리지 임시 전략(로컬 임시폴더 또는 S3 호환 저장소)
- 완료 조건:
  - 팀 외부 사용자가 URL로 업로드-처리-다운로드 1회 성공

## Week 4 - 안정화 + 데모 패키징
목표: 발표/포트폴리오 가능한 형태로 마무리
- 작업:
  - 에러 핸들링(파일 형식/크기 제한, 타임아웃) 보강
  - README/아키텍처 다이어그램/배포 문서 정리
  - 3분 데모 시나리오 작성(전/후 비교)
  - 태그 릴리즈(`v0.1.0`)
- 완료 조건:
  - 신규 팀원이 문서만 보고 로컬 실행 성공
  - 데모 영상/스크린샷 확보

## 5. 배포 트랙 vs 학습 트랙 (병렬 운영)
### 원칙
- 메인 트랙은 `배포`로 유지한다. (프로젝트가 멈추지 않도록)
- 학습은 `별도 실험 트랙`으로 진행하고, 검증 후에만 서비스에 반영한다.
- 모델 교체는 감으로 하지 않고, 동일 데이터셋 기준 비교 후 결정한다.

### 트랙 A: 배포 트랙 (Main)
- 현재 pretrained 모델 기반으로 기능 완성 및 운영 안정화
- KPI:
  - 처리 성공률
  - 처리시간(p50/p95)
  - 익명화 누락 건수
- 산출물:
  - 배포 URL
  - 운영 문서(실행/롤백/로그 확인)

### 트랙 B: 학습 트랙 (Experiment)
- 데이터 정리 -> 파인튜닝 -> 경량화 -> 비교 평가 순서
- KPI:
  - 정확도(누락/오검출)
  - 속도(FPS/총 처리시간)
  - 모델 크기/메모리
- 산출물:
  - 실험 리포트(기존 모델 대비 표)
  - 배포 반영 가능 모델 아티팩트(ONNX 등)

### 반영 게이트 (서비스 적용 조건)
- 기준 모델 대비 정확도 동등 이상
- 처리속도 또는 비용 중 하나라도 개선
- 장애율 증가 없음(메모리/타임아웃)

## 6. 릴리즈 단계 (권장)
1. `v0.1` (배포 우선): pretrained 모델로 동영상 서비스 배포
2. `v0.2` (실험 병행): 학습 모델을 옵션 플래그로 병행 운영(A/B 비교)
3. `v1.0` (안정 전환): 검증 통과 모델을 기본 모델로 승격

## 7. 백로그 우선순위 (MoSCoW)
### Must
- 비디오 업로드/처리/다운로드
- 기본 성능 개선 1회 이상
- 실제 배포 URL

### Should
- 처리 큐(간단한 job status)
- 로그 정리 및 에러 메시지 개선

### Could
- 웹캠 실시간 처리(브라우저 기반)
- 사용자별 기준 얼굴 프로필 관리

### Won't (이번 라운드)
- 네이티브 앱 스토어 배포
- 멀티모델 자동 선택

## 8. 기술 선택 가이드 (가볍게)
- 현재 코드베이스 기준 추천:
  - 백엔드: `FastAPI` 유지 (이미 존재)
  - 프론트: 최소 UI(React 또는 템플릿 기반)
  - 배포: 관리형 PaaS 우선(서버 세팅 시간 절감)
- 판단 원칙:
  - “새 기술 도입”보다 “완료 가능성” 우선
  - 1주 내 데모 불가한 선택지는 보류

## 9. 리스크와 대응
- 리스크: 비디오 처리시간 과다
  - 대응: 리사이즈/프레임 스킵 기본값 제공
- 리스크: 배포 환경에서 모델 로딩 실패
  - 대응: 헬스체크 + 모델 경로 검증 + 시작 로그
- 리스크: 일정 지연
  - 대응: 실시간 기능 제외, 파일 업로드 처리만 고정

## 10. 운영 규칙 (사이드 프로젝트용)
- 주 2회, 30분 짧은 스탠드업
- PR은 작게(300 lines 내외), 리뷰 당일 처리
- “동작하는 것 먼저, 예쁘게는 나중”

## 11. 첫 주 바로 실행할 액션 아이템
1. 비디오 처리 함수 시그니처 확정 및 이슈 생성
2. 샘플 영상 2개 준비(짧은/긴 영상)
3. 성능 측정 기준표 템플릿 생성(FPS, 총시간, 해상도)
4. 배포 타겟 1개 확정(Render 또는 Railway)
